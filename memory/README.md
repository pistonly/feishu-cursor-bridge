# 项目记忆（`memory/`）

本目录是**顶层集成仓库**的持续开发记忆层：跨模块背景、长期有效结论、便于接手的当前状态。不写命令流水账、聊天逐字稿，也不把只属于子模块内部的细节堆在这里。

## 命名

- 新建文件优先：`YYYY-MM-DD-HHMM-topic-stage.md`
- `topic`：稳定、可检索的短语，同一问题线不要随意改名
- `stage`：建议用小词表，例如 `triage`、`blocker`、`fix`、`verify`、`recovery`、`summary`

## 同一天、同一 topic、同一 stage

若仍属于**同一条问题线**的补充，**优先在已有文件末尾续写**，不要重复开新文件。

在下列情况再**新建**文件：`topic` 明显变化；进入新的 `stage`；已形成可单独引用的长期结论；续写会混淆阶段边界。

## 推荐结构（中文）

文件开头可加简短元信息，正文推荐小节：

```markdown
# <标题>

- status: active | resolved | blocked
- related_modules: ...
- related_memory:
- supersedes:

## 背景
## 关键结论
## 影响范围（或：涉及的主要变动）
## 关联版本（推荐）
## 当前状态
## 后续建议
```

## 关联版本

写清楚结论对应的代码状态，便于回溯：

- 顶层仓库：`git rev-parse HEAD`，并注明是否有未提交改动（dirty）
- 若涉及子模块，分别记录子模块的 `HEAD`
- 结论若只对应未提交工作区，须写明，避免误以为已进某次 commit
